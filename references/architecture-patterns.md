# 工程架构模式

## 方案确认后的最小分层

推荐数据流：

```text
输入设备 -> 采集 -> 视觉观测 -> 坐标/状态 -> 通信或执行 -> 反馈/日志
```

模块只在真实职责、独立验证风险或复用需要时存在。没有通信、UI、模型、闭环或独立风险时，不创建空模块。

## 状态和数据契约

每个跨模块结果至少说明：来源、时间戳、坐标系、单位、有效性、置信度/质量、超时和失败动作。不要把一个裸坐标或布尔值当作完整闭环状态。

常见状态：`INIT`、`SEARCH`、`TRACKING`、`CONFIRMED`、`LOST`、`SAFE_STOP`。按题目需要裁剪，不为统一格式强行加入状态。

## 可替换点

把以下变化隔离在明确边界：摄像头后端、传统视觉/模型算法、标定参数、目标选择、串口传输、下位机控制和执行器。先确定接口语义，再替换实现；不要让摄像头设备节点、串口帧或模型输出字段散落在主循环中。

## 独立验证顺序

先验证输入，再验证算法，再验证坐标，再验证通信/执行，最后验证闭环和题目指标。每一步都保留最小日志和失败输出。

## 反模式

- 在未确认系统/API 时直接生成完整工程；
- 把桌面 OpenCV 结果当成 RK3566 实测；
- 为“工程化”创建未执行的通信、显示、模型和测试文件；
- 在视觉失效时继续输出旧坐标而没有时间戳和超时；
- 同时改变阈值、坐标、协议和控制增益，导致无法归因。
